# 16-Bit Adder Designs

This repository contains two VHDL implementations of a 16-bit binary adder.
The goal is to make the same arithmetic interface available through two architectures.
The first design uses a ripple-carry structure built from one-bit full adders.
The second design uses carry-lookahead logic arranged as four 4-bit blocks.
Both designs are packaged as independent Xilinx Vivado projects.

## Project Scope

The repository focuses on synthesizable combinational digital logic.
Each adder accepts two 16-bit operands and one external carry input.
Each adder returns a 16-bit sum and a final carry output.
The source is written in VHDL using IEEE `STD_LOGIC_1164`.
Testbenches use `NUMERIC_STD` for readable numeric reporting.
The project is source-oriented and excludes generated Vivado artifacts.

## Repository Layout

`ripple_adder_16bit.xpr` opens the ripple-carry Vivado project.
`carry_lookahead_adder_16bit.xpr` opens the carry-lookahead Vivado project.
`ripple_adder_16bit.srcs/` contains ripple-carry source and simulation files.
`carry_lookahead_adder_16bit.srcs/` contains carry-lookahead source and simulation files.
`.gitignore` keeps generated simulation, synthesis, and editor files out of Git.
`README.md` provides project documentation and usage notes.

## Ripple-Carry Adder

The ripple design is implemented in `ripple_adder_16bit.vhd`.
It instantiates sixteen copies of `one_bit_full_adder`.
Bit 0 receives the external `carry_in` signal.
Each following bit receives the carry generated by the previous bit.
The final carry in the internal chain becomes `carry_out`.
This architecture is simple, direct, and easy to inspect.
Its main tradeoff is that carry propagation depends on the full bit width.

## Carry-Lookahead Adder

The carry-lookahead design is implemented in `cla_adder_16bit.vhd`.
It divides the 16-bit operation into four 4-bit adder blocks.
Each 4-bit block reports a group propagate signal and a group generate signal.
A top-level lookahead unit computes the carry inputs for the 4-bit groups.
The lower-level lookahead unit also computes bit-level carries inside each group.
This reduces the amount of serial carry propagation in the design.
The structure is more modular than the ripple design but uses more logic.

## Common Interface

Both top-level adders use the same external signal names.
`input_a` is the first 16-bit operand.
`input_b` is the second 16-bit operand.
`carry_in` is the optional incoming carry bit.
`sum_out` is the 16-bit result.
`carry_out` is the carry produced beyond bit 15.
This shared interface makes the two architectures easy to compare.

## Source Modules

`one_bit_full_adder.vhd` computes sum, propagate, and generate for one bit.
`ripple_adder_16bit.vhd` connects sixteen full adders in a carry chain.
`carry_lookahead_unit_4bit.vhd` computes carries from propagate and generate inputs.
`cla_adder_4bit.vhd` combines four full adders with a 4-bit lookahead unit.
`cla_adder_16bit.vhd` combines four 4-bit CLA blocks into one 16-bit adder.
The files are intentionally named to describe their role in the hierarchy.

## Verification

Each project includes a VHDL testbench under its `sim_1/new` directory.
`tb_ripple_adder_16bit.vhd` verifies the ripple-carry implementation.
`tb_cla_adder_16bit.vhd` verifies the carry-lookahead implementation.
The testbenches apply representative hexadecimal input vectors.
They check normal additions, carry-in behavior, overflow, and boundary values.
Assertions report an error if a computed sum or carry does not match expectation.
Informational reports identify which test vectors passed during simulation.
The default wait period between test vectors is 20 ns.

## Running in Vivado

Open Vivado and select one of the `.xpr` files from this repository.
Use the Sources view to inspect the VHDL hierarchy.
Use Run Simulation to execute the included behavioral testbench.
Use Run Synthesis to check that the selected adder is synthesizable.
Keep the matching `.srcs` directory with the project file when moving the project.
Generated Vivado outputs are ignored so the repository stays clean.

## Design Notes

The arithmetic is unsigned binary addition at the bit-vector level.
The designs are combinational and do not require a clock or reset.
Overflow is represented by `carry_out`, not by expanding `sum_out`.
The ripple-carry design is useful as a readable baseline implementation.
The carry-lookahead design is useful for comparing hierarchy and carry optimization.
Both versions can be connected to larger datapaths or ALU experiments.
